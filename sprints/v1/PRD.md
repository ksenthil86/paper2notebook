# Sprint v1 — PRD: Research Paper to Colab Notebook

## Overview
Build a web application where researchers upload a research paper PDF and receive a production-quality, deeply structured `.ipynb` Google Colab notebook. The notebook implements the paper's algorithms and methodology as a research-grade tutorial — with realistic synthetic data, rigorous explanations, and runnable experiments — powered by OpenAI's reasoning model using the researcher's own API key. The app features a beautiful dark UI inspired by arcprize.org, with live status updates during generation to keep the user engaged.

## Goals
- Researcher enters their OpenAI API key, uploads a PDF, and clicks Generate — zero other setup required
- Backend uses gpt-5.4 (OpenAI reasoning model) with a structured prompt to produce a research-grade notebook
- Generated notebook is non-trivial: realistic synthetic data, full algorithm implementations, visualizations, and interpretive commentary suitable for researchers at top labs
- User can download the `.ipynb` directly **and** open it in Google Colab with one click (via GitHub Gist)
- While the notebook is being generated, the UI streams status messages so the user stays engaged (e.g. "Parsing paper...", "Identifying algorithms...", "Writing implementation cells...", "Building notebook...")
- UI/UX is beautiful, dark, and technical — inspired by arcprize.org (dark background, clean typography, monospace accents, minimal but polished)

## User Stories
- As a researcher, I want to upload a PDF and get a runnable notebook so I can replicate the paper's methodology without manually implementing it
- As a researcher, I want to use my own OpenAI API key so I control costs and data privacy
- As a researcher, I want to open the notebook directly in Google Colab so I can run it immediately without any upload steps
- As a researcher at a top lab, I want the notebook to be production-quality — with rigorous math commentary, realistic synthetic data, and proper experiment structure — not a toy demo
- As a researcher, I want to see live progress updates while waiting so I know the system is working and what it's doing

## Technical Architecture

**Frontend**: React (Vite) + custom CSS (arcprize.org-inspired dark theme)
**Backend**: FastAPI (Python)
**PDF Parsing**: pdfplumber
**Notebook Generation**: OpenAI gpt-5.4 (reasoning model) via structured prompt
**Notebook Assembly**: nbformat
**Open in Colab**: GitHub Gist API (user optionally provides GitHub PAT, or we create anonymous gist)
**Progress Updates**: Server-Sent Events (SSE) stream from backend to frontend

```
┌──────────────────────────────────────────────┐
│              React Frontend                  │
│                                              │
│  [OpenAI API Key]  [GitHub Token (optional)] │
│  [PDF Upload drop zone]                      │
│  [Generate button]                           │
│                                              │
│  ── Generating ──────────────────────────    │
│  ✓ Parsing PDF...                            │
│  ✓ Identifying algorithms...                 │
│  ⟳ Writing implementation cells...           │
│                                              │
│  [Download .ipynb]  [Open in Colab ↗]        │
└───────────────┬──────────────────────────────┘
                │ POST /generate (multipart)
                │ + GET /status/{job_id} (SSE)
┌───────────────▼──────────────────────────────┐
│              FastAPI Backend                 │
│                                              │
│  1. pdfplumber      → extract full text      │
│  2. gpt-5.4         → structured cell JSON   │
│  3. nbformat        → build .ipynb           │
│  4. GitHub Gist API → upload + get gist URL  │
│  5. Return .ipynb download + colab_url       │
└──────────────────────────────────────────────┘
```

**Data flow:**
1. User submits PDF + OpenAI key (+ optional GitHub token) via form
2. Backend spawns background job, returns `job_id` immediately
3. Frontend opens SSE stream to `GET /status/{job_id}` — backend emits status events as each phase completes
4. Backend phases (each emits SSE event):
   - `parsing` → pdfplumber extracts full text
   - `analyzing` → GPT call: extract paper title, algorithms, methods, math notation
   - `generating` → GPT call: generate full notebook cell JSON (see Notebook Structure below)
   - `assembling` → nbformat builds `.ipynb` from cell JSON
   - `uploading` → upload to GitHub Gist (if token provided), get Colab-compatible URL
   - `done` → emit download URL + colab URL
5. Frontend shows each phase in real-time, then reveals Download + Open in Colab buttons

**Notebook Structure (generated by gpt-5.4):**

The model is prompted to produce a research-grade notebook with this structure:
```
1. Title cell        — paper title, authors, venue, abstract
2. Setup cell        — !pip install all dependencies
3. Imports cell      — all imports with version pins
4. Overview cell     — markdown: paper's core contribution, key equations (LaTeX)
5. Per-algorithm:
   a. Theory cell    — markdown: math formulation, assumptions, complexity
   b. Impl cell      — full Python implementation (not pseudocode)
   c. Data cell      — realistic synthetic data generation (NOT iris/MNIST)
   d. Experiment cell — run the algorithm, log metrics
   e. Viz cell       — matplotlib/seaborn visualizations of results
6. Comparison cell   — compare to baseline method(s)
7. Discussion cell   — markdown: limitations, extensions, connection to related work
```

The prompt explicitly instructs the model:
- Use realistic synthetic data that mirrors the problem domain (e.g. for a graph neural network paper: generate synthetic graphs with community structure)
- Include proper error handling, type hints, and docstrings
- Write LaTeX math in markdown cells
- Generate production-quality Python, not demo snippets

**Open in Colab mechanism:**
- If user provides GitHub token: POST to `api.github.com/gists` to create a public gist with the `.ipynb` content → construct `https://colab.research.google.com/gist/{username}/{gist_id}` URL
- If no GitHub token: Download-only (no Colab link shown); user can upload manually

## UI/UX Design (arcprize.org-inspired)

**Theme**: Dark, technical, minimal
- Background: `#0a0a0b` (near black)
- Surface: `#111113` (card/panel bg)
- Border: `#222226`
- Primary text: `#f0f0f0`
- Accent: `#e8c547` (yellow — arcprize brand color)
- Monospace font for inputs, filenames, status messages: `JetBrains Mono` or `IBM Plex Mono`
- Body font: `Inter`

**Layout**: Single-page, centered, max-width 680px
- Header: App name + one-line description
- Form card: API key field → GitHub token field (collapsible/optional) → PDF drop zone → Generate button
- Progress panel: appears after submit, shows live status with animated dots/checkmarks
- Result panel: appears on completion — Download button + Open in Colab button (if available)

**Status messages** (displayed sequentially during generation):
1. "Parsing PDF and extracting text..."
2. "Identifying algorithms, methods, and key equations..."
3. "Generating implementation — this takes a moment with reasoning models..."
4. "Assembling notebook cells..."
5. "Uploading to GitHub Gist..." (if token provided)
6. "Done! Your notebook is ready."

## Out of Scope (v2+)
- User authentication and accounts
- Usage tracking and billing
- Storing generated notebooks server-side
- Section selection / partial paper processing
- Notebook preview in the browser
- Multiple paper comparison
- Google Drive / Google Colab direct API integration (without Gist)
- Streaming the notebook generation token-by-token

## Dependencies
- Greenfield project — no prior sprint
- Researcher must supply a valid OpenAI API key with gpt-5.4 access
- "Open in Colab" requires an optional GitHub Personal Access Token (gist scope)

## Notes
- gpt-5.4 is specified as the target model; if unavailable, fall back to `o3` or `gpt-4.1` in that order
- Generation time will be 30–120 seconds depending on paper length and model reasoning time; SSE progress is critical for UX
- Notebooks should target Python 3.10+, Colab runtime
